# Evidence Note — Capacity-Dependent Readout and a Local Verifier-Log Observation

This note states only what the eBPF artifact and its verifier trace directly
establish. It does not infer a joined verifier cell, singleton-state abstraction
equivalence, or the standard abstract-interpretation completeness equation from
an unresolved branch.

## 1. Bound Evidence

The authoritative files are generated by one run and bound by
`results/nand.provenance.json`:

- the preserved BPF object and userspace harness under
  `results/variants/<run_id>/nand/`;
- `results/env.json`;
- `results/nand.verifier.log`;
- `results/nand.wm_nand.xlated.txt`; and
- the normal NAND, full-adder, sampled 32-bit, and exhaustive 8-bit JSONL files.

The schema-v2 manifest hashes every file. `scripts/audit_results.py` rejects
legacy unbound manifests, mixed run IDs, missing files, and digest mismatches.

## 2. Concrete Runtime Observation

The gate map is an ordinary, non-LRU `BPF_MAP_TYPE_HASH` with logical
`max_entries = 2`, default preallocation, keys `S,A,B`, and `BPF_ANY`
updates. It is not a per-CPU hash map. Kernel-internal per-CPU allocation
freelists or spare elements are implementation details and do not enlarge the
map's logical `max_entries`.

After deleting `S,A,B`, the program inserts sentinel `S`, applies the
first input-conditioned update, then performs the second input-conditioned
update. The eBPF program copies that second update's complete 64-bit return
pattern into `TAPE[SECOND_UPDATE_RAW_IDX]` and sets a validity flag. Userspace
decodes it as signed `int64_t` and writes three fields on every result row:

- `second_update_observed`;
- `second_update_raw_ret`; and
- `second_update_errno`, derived as `-raw_ret` for a negative return.

For normal one-gate NAND, the audited relation is:

| input | selected second operation | raw return | output `[ret == 0]` |
|---|---|---:|---:|
| `00`, `01`, `10` | update existing or insert with room | `0` | 1 |
| `11` | insert fresh key at logical capacity | `< 0` | 0 |

The exact negative errno is a recorded fact of that run. The proof and auditor
do **not** hard-code `E2BIG` or any other numeric errno as a cross-kernel
premise. Both capacity-removing ablations record return zero. The explicit
Boolean baseline has no capacity-probe helper and therefore records
`observed=false` with null return/errno fields.

## 3. Local Verifier-Log Observation

The hash-bound verifier log supplies the following build-specific local
observation:

1. the second input-conditioned `bpf_map_update_elem` returns a scalar;
2. that return is preserved for the output zero-test;
3. the verifier explores both successors of `ret == 0`; and
4. the xlated bytecode copies the same raw return to the evidence slot and
   derives `IDX_NAND_OUT` from its zero predicate.

Instruction numbers and map IDs are intentionally not copied here because
instrumentation and compiler versions change them. Review the bound log and
xlated file together. `xlated` is verifier-processed eBPF bytecode, not
architecture-specific JIT-native machine code.

## 4. What the Evidence Licenses

The direct conclusion is local and empirical:

> In the accepted normal NAND program, concrete hash-map capacity semantics
> determine the second update's zero/negative return and therefore the output
> bit. At the instrumented frontier, the captured verifier log represents that
> return as a scalar and displays both successors of its zero-test.

The runtime behavior is documented hash-map capacity and helper-return behavior.
If a future `Report_log` interpretation established that an admissible report
omits this distinction while satisfying the required coverage and transfer-
soundness premises, the resulting classification would be contract-shape-
induced and contract-relative: tracking occupancy or forbidding the helper
could remove that readout. The current artifact does not establish those
report-level premises.

This supports calling the log evidence a *local scalar-unresolved observation*.
It does not by itself establish any of the following stronger claims:

- a computed report cell `a#` that jointly covers two specified executions;
- `α({σ0}) = α({σ1})` for two individual concrete singleton states;
- equality of pointwise abstract traces;
- failure of the standard completeness equation
  `α ∘ f = f# ∘ α`; or
- absence of every possible relational certificate for the full input/output
  graph.

A formal result that needs those claims must separately define a `Report_log`
interpretation and extractor for the analyzer's computed state `a#`, show the
relevant executions belong to `γ(a#)`, define the certificate vocabulary, and
prove the required local coverage, transfer-soundness, adequacy, or soundness
property. The artifact does not substitute branch uncertainty for that proof.

## 5. Reproduction and Audit

Run:

```sh
./scripts/run_kernel_suite.sh
make verify
```

The full audit independently recomputes all truth tables and sums, requires raw
return `0` for successful probes and a negative return for the normal
at-capacity case, checks errno consistency, and validates every provenance
binding. Historical results without the three helper-observation fields fail
with an explicit request to rerun; the tool never synthesizes raw returns for
old rows.
